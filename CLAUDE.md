# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repo Is

A conference demo showing the full agentregistry lifecycle: install the platform, build an MCP server, package a skill, compose an agent, publish everything to a registry, deploy to Kubernetes, and connect to Claude Code. The demo is structured as a 7-act storyboard (see `STORYBOARD.md`).

The repo is organized by demo act, not by application layer:

- `00-Install/` — Platform prerequisite commands (arctl, kind, kagent, agentgateway)
- `01-MCP/ops-server/` — FastMCP-based MCP server (Python 3.12, `uv` for deps)
- `02-Agents/platform-ops-agent/` — Google ADK agent (Python 3.13, `uv` for deps)
- `03-Skills/platform-best-practices/` — Skill definition (SKILL.md + reference docs)

## Build & Test Commands

### MCP Server (`01-MCP/ops-server/`)

```bash
cd 01-MCP/ops-server

# Install dependencies
uv sync

# Run tests (all)
uv run pytest

# Run a single test file
uv run pytest tests/test_get_service_health.py

# Run a single test
uv run pytest tests/test_tools.py::TestToolLoading::test_server_initialization

# Lint
uv run ruff check src/ tests/
uv run black --check src/ tests/

# Type check
uv run mypy src/

# Run server locally (stdio mode)
uv run python src/main.py

# Run server locally (HTTP mode)
uv run python src/main.py --transport http --host localhost --port 3000

# Build container
arctl mcp build . --image ops-server

# Publish to registry
arctl mcp publish . --type oci --package-id ops-server
```

### Agent (`02-Agents/platform-ops-agent/`)

```bash
cd 02-Agents/platform-ops-agent

# Install dependencies
uv sync

# Run locally via docker-compose (needs OPENAI_API_KEY set)
docker compose up

# Build container
arctl agent build . --push --platform linux/amd64

# Run via arctl
arctl agent run platform-ops-agent
```

## Architecture

### MCP Server Tool Pattern

Tools live in `01-MCP/ops-server/src/tools/`, one file per tool. Each tool file imports the global `mcp` instance from `core.server` and registers itself via the `@mcp.tool()` decorator. The `DynamicMCPServer` in `core/server.py` auto-discovers all `.py` files in the tools directory at startup (excluding `__init__.py`). If any tool fails to load, the server exits immediately.

All tool data is hardcoded mock data for the demo (e.g., payments service is always degraded with 12.5% error rate).

Transport mode is controlled by `--transport` flag or `MCP_TRANSPORT_MODE` env var (values: `stdio`, `http`).

### Agent Composition

The agent (`02-Agents/platform-ops-agent/`) is built with Google ADK (`google-adk`). It composes three registry artifacts defined in `agent.yaml`:
- **MCP server** (`ops-server`) — provides tools like `get_service_health`, `list_deployments`, `get_logs`
- **Skill** (`platform-best-practices`) — SLA tiers, escalation procedures, team ownership
- **Prompt** (`strict-operator`) — formal operator persona with behavior rules

`mcp_tools.py` and `prompts_loader.py` are autogenerated by `arctl` — do not edit them directly. They load runtime config from `/config/mcp-servers.json` and `/config/prompts.json` respectively (or from files adjacent to the module for local dev).

The agent uses OpenAI's `gpt-4o` as the model provider (configurable via `MODEL_PROVIDER` and `MODEL_NAME` env vars).

### Skill Structure

Skills are Markdown-based knowledge packages. `03-Skills/platform-best-practices/SKILL.md` contains structured operational knowledge (SLA tiers, health thresholds, deployment standards, team ownership, escalation procedures). Skills are built into Docker images and published to the registry.

## Key Environment Variables

- `OPENAI_API_KEY` — Required for the agent (model provider)
- `MCP_TRANSPORT_MODE` — `stdio` or `http` (MCP server transport)
- `HOST` / `PORT` — HTTP bind address for MCP server (defaults: `localhost`, `3000`)
- `MCP_TERMINATE_ON_CLOSE` — Set `false` for stateless MCP servers
- `OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` — Optional OpenTelemetry traces endpoint

## Registry & Deployment

The registry runs locally via `arctl daemon start` at `localhost:12121`. Deployment targets a kind cluster named `agentregistry`. The CLI tool `arctl` handles scaffolding, building, publishing, and deploying all artifact types (MCP servers, skills, prompts, agents).
